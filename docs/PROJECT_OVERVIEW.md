# Обзор проекта

## Назначение

`Eng_estimate` - локальное веб-приложение для оценки эффективности инжиниринга по проектам. Оно переносит рабочую логику из Excel-файла `Engineering Efficiency Measurement.xlsx` в интерфейс, пригодный для ежедневной работы.

Приложение позволяет:

- создавать и хранить карточки проектов;
- считать плановый показатель эффективности `PLAN`;
- считать фактический показатель эффективности `FACT`;
- видеть структуру расчета без открытия Excel;
- агрегировать статистику по сохраненным проектам.

## Источник требований

Проект опирается на два основных артефакта:

1. `Текст ТЗ.docx`
2. `Engineering Efficiency Measurement.xlsx`

Текст ТЗ определяет функциональный сценарий. Excel определяет факторы, допустимые значения, коэффициенты, границы и итоговые формулы.

## Основной пользователь

Целевой пользователь - специалист, который:

- ведет проекты;
- заполняет параметры оценки;
- сравнивает плановые и фактические показатели;
- формирует локальную базу проектов для анализа.

## Пользовательский сценарий

### Создание проекта

Пользователь создает новый проект и заполняет:

- наименование;
- код проекта;
- группу проектов;
- стадию;
- дату начала;
- дату окончания;
- руководителя проекта;
- ГИПа;
- ответственного за проектирование.

### Работа с PLAN

На вкладке `PLAN` пользователь выбирает qualitative-оценку для факторов из Excel-модели. Приложение:

- показывает факторы из `input I`;
- переводит qualitative-значение в числовое;
- считает все промежуточные показатели;
- возвращает итоговый индекс `PLAN`.

### Работа с FACT

На вкладке `FACT` пользователь вводит фактические значения по показателям. Приложение:

- показывает структуру показателей из листа `engineering efficiency FACT`;
- показывает границы и веса;
- нормализует значения;
- возвращает итоговый индекс `FACT`.

### Работа со статистикой

На вкладке `Статистика` пользователь фильтрует проекты:

- по группе;
- по периоду;
- по специалисту.

Приложение строит агрегаты:

- количество проектов;
- средний `PLAN`;
- средний `FACT`;
- средний разрыв `FACT - PLAN`, если доступны оба индекса;
- распределение по стадиям.

## Что входит в текущий MVP

- локальное хранение данных;
- один пользователь;
- web UI;
- чтение Excel при старте;
- объяснимый breakdown по результатам;
- базовые API endpoints.

## Что не входит

- авторизация;
- многопользовательский режим;
- общая база данных;
- внешние интеграции;
- редактирование Excel в браузере;
- сложная отчетность и BI.

## Почему Excel не используется как runtime-движок

Excel выступает источником модели, но не механизмом исполнения на сервере. Это нужно, чтобы:

- расчеты были тестируемыми;
- API было стабильным;
- приложение работало без Excel-клиента;
- изменения модели можно было контролировать через код и тесты.
