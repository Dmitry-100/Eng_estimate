# Архитектура

## Принцип

Проект построен как легкое локальное приложение, но с разделением ответственности между:

1. HTTP-слоем
2. парсером Excel-модели
3. расчетной логикой
4. хранилищем проектов
5. статистикой
6. UI

## Структура каталогов

```text
app.py
eng_efficiency/
  workbook.py
  calculator.py
  storage.py
  statistics.py
data/
  projects.json
templates/
  index.html
static/
  css/styles.css
  js/script.js
tests/
  test_app.py
docs/
  ...
```

## Ответственность модулей

### `app.py`

Точка входа Flask.

Отвечает за:

- создание приложения;
- загрузку workbook-модели;
- инициализацию хранилища;
- регистрацию HTTP endpoints;
- валидацию входных HTTP данных;
- трансляцию ошибок в JSON-ответы.

### `eng_efficiency/workbook.py`

Парсер `.xlsx`.

Отвечает за:

- чтение workbook XML;
- извлечение shared strings;
- поиск листов;
- построение `WorkbookModel`;
- извлечение факторов `PLAN`;
- извлечение показателей `FACT`;
- извлечение коэффициентов, весов и линейных функций нормализации.

### `eng_efficiency/calculator.py`

Чистая бизнес-логика расчета.

Отвечает за:

- `calculate_plan`;
- `calculate_fact`;
- нормализацию показателей;
- формирование breakdown по расчету;
- ошибки валидации уровня домена.

### `eng_efficiency/storage.py`

Локальный репозиторий проектов.

Отвечает за:

- создание файла хранения;
- чтение списка проектов;
- чтение одного проекта;
- создание и обновление записей;
- атомарную запись JSON.

### `eng_efficiency/statistics.py`

Агрегация данных.

Отвечает за:

- фильтр по группе;
- фильтр по специалисту;
- фильтр по периоду;
- расчет агрегатов;
- подготовку компактного статистического ответа.

### `templates/index.html`

HTML-каркас интерфейса:

- список проектов;
- карточка проекта;
- вкладки `PLAN`, `FACT`, `Статистика`;
- контейнеры под вывод результатов.

### `static/js/script.js`

Клиентская orchestration-логика:

- загрузка конфигурации;
- загрузка проектов;
- сохранение проекта;
- отправка расчетов;
- переключение вкладок;
- рендер результатов.

### `static/css/styles.css`

Оформление интерфейса и адаптивное поведение.

## Потоки данных

## Старт приложения

1. Flask стартует.
2. Загружается `Engineering Efficiency Measurement.xlsx`.
3. Из workbook строится `WorkbookModel`.
4. Инициализируется `ProjectStore`.
5. UI запрашивает `/api/config` и `/api/projects`.

## Поток PLAN

1. UI получает список факторов.
2. Пользователь выбирает оценки.
3. UI вызывает `POST /api/projects/<id>/plan`.
4. Backend считает `PLAN`.
5. Результат сохраняется в проекте.
6. UI показывает итог и breakdown.

## Поток FACT

1. UI получает список FACT-показателей.
2. Пользователь вводит значения.
3. UI вызывает `POST /api/projects/<id>/fact`.
4. Backend считает `FACT`.
5. Результат сохраняется в проекте.
6. UI показывает итог и breakdown.

## Поток статистики

1. UI вызывает `GET /api/statistics`.
2. Backend читает проекты из хранилища.
3. Применяются фильтры.
4. Возвращаются summary, распределение по стадиям и список проектов.

## Почему выбран JSON storage

Для MVP это дает:

- нулевую инфраструктуру;
- простую переносимость;
- прозрачность данных;
- минимальную стоимость поддержки.

Если проект вырастет, естественный следующий шаг - перейти на SQLite, сохранив границы модулей `storage.py` и `statistics.py`.
